Data Modeling Foundations

Module 3 | Day 1

Master the fundamentals of data modeling in Power BI. Learn star schema design, table relationships, and cardinality to build efficient, scalable analytical models.

Learning Objectives

Star Schema Design

The star schema is the foundational design pattern for analytical data models. It consists of:

Fact Tables
  • Contain measurable events (sales, orders)
  • Store numeric values (amounts, quantities)
  • Include foreign keys to dimensions
  • High row count, few columns
  • Example: Sales, Inventory, Orders
Dimension Tables
  • Provide context for facts
  • Store descriptive attributes
  • Lower row count, more columns
  • Used for filtering and grouping
  • Example: Products, Customers, Dates
Why Star Schema?

Star schema optimizes query performance, simplifies DAX calculations, and makes models intuitive. It's the industry standard for OLAP systems and the recommended approach for Power BI.

Table Relationships

Relationships connect tables and enable filter propagation across your data model.

-- Relationship Structure: From Table (Many Side) → To Table (One Side) Sales[ProductKey] → Products[ProductKey] Sales[CustomerKey] → Customers[CustomerKey] Sales[DateKey] → Calendar[DateKey]
One-to-Many (1:*)

Most common. One dimension record relates to many fact records. Default cross-filter: Single direction.

Many-to-One (*:1)

Inverse of One-to-Many. Same behavior, different direction of relationship creation.

One-to-One (1:1)

Rare. Both sides have unique values. Typically used for extending tables or security scenarios.

Cardinality in Detail

Cardinality defines how rows in one table relate to rows in another table.

One-to-Many Example
Customers Table (One Side): CustomerKey | CustomerName 1 | Contoso 2 | Fabrikam Sales Table (Many Side): SalesKey | CustomerKey | Amount 1001 | 1 | 500 1002 | 1 | 750 1003 | 2 | 300 One customer can have many sales transactions.

Key Rule: The "One" side must contain unique values. The "Many" side can have duplicate values. Power BI enforces this through referential integrity.

Cross-Filter Direction

Controls how filters propagate through relationships.

Single Direction →

Default and Recommended

  • Filters flow from "One" to "Many"
  • Dimension filters affect facts
  • Better performance
  • Easier to understand and debug
Both Directions ↔

Use Sparingly

  • Filters flow in both directions
  • Can cause ambiguity
  • Performance overhead
  • Only for specific scenarios
When to Use Both?

Bi-directional filtering is needed for many-to-many relationships, role-playing dimensions with inactive relationships, or when dimension tables must filter each other. However, use with caution.

Best Practices

✓ Do This
  • Always use star schema design
  • Create unique keys for relationships
  • Use single-direction filters (default)
  • Keep dimension tables narrow
  • Create a separate Calendar table
  • Document your model structure
✗ Avoid This
  • Circular relationships
  • Too many bi-directional filters
  • Complex chain relationships
  • Mixing fact and dimension data
  • Storing aggregates in facts
  • Using calculated columns for keys

Key Takeaways

  • Star schema is the foundation of good Power BI models
  • Fact tables store measurable events; dimension tables provide context
  • One-to-Many is the most common cardinality type
  • Single-direction cross-filtering is preferred for performance
  • Proper relationships enable accurate DAX calculations
  • Good data modeling is 80% of Power BI success
